## Finding 1 — mintsysadm.py is a grub write-back trap

**File:** `/usr/lib/python3/dist-packages/mintsysadm/mintsysadm.py`
(or equivalent path on installed system)

The legitimate mintsysadm script has been injected with a **duplicate save function block**
that fires after the real one. Identified by:
- Chinese character line `什 吧二0：` mid-code (injected marker)
- Duplicate `GRUB_TIMEOUT_STYLE`, `GRUB_TIMEOUT`, `GRUB_CMDLINE_LINUX_DEFAULT` definitions
- The save function expands `$GRUB_CMDLINE_LINUX_DEFAULT` — rootkit's cage flags already
  live in that variable, so every time user opens System Admin and saves, they write
  rootkit's flags back into grub themselves

**Effect:** Any attempt to strip cage flags via the GUI tool silently restores them.

**Headshot:** Replace with clean version from `github.com/linuxmint/mintsysadm` before
touching grub via GUI. The injected second block is the actual target.

## Finding 2 — ubiquity launch broadcasts rootkit's alert signal

**From:** `ubiquity.desktop` exec line:

-preserve-env=DBUS_SESSION_BUS_ADDRESS,XDG_DATA_DIRS,XDG_RUNTIME_DIR,GTK_THEME


The moment ubiquity launches normally, it preserves and broadcasts `DBUS_SESSION_BUS_ADDRESS`.
Rootkit's AT-SPI bus is watching that exact variable — this is the trigger.

**Headshot:** Launch ubiquity without preserving that variable:


unset DBUS_SESSION_BUS_ADDRESS WEBKIT_DISABLE_COMPOSITING_MODE=1 ubiquity gtk_ui


No pkexec, no env preservation, no dbus signal. Rootkit is blind for longer.

## Finding 3 — AT-SPI is rootkit's covert graphical surveillance channel

**From:** auth log + dbus command line in task manager:

unix:path=/run/user/29999/at-spi/bus_0


AT-SPI (accessibility bus) running under uid 29999 (OEM user). Rootkit uses the
accessibility layer as a covert dbus channel — every window, every keystroke, every
UI element goes through it. This is how it monitors graphical sessions without needing
VT7 active.

**Headshot:** Before launching graphical installer:


systemctl --user stop at-spi2-registryd systemctl --user mask at-spi2-registryd unset AT_SPI_BUS

systemctl --user stop at-spi2-registryd systemctl --user mask at-spi2-registryd unset AT_SPI_BUS


Kills the surveillance channel before ubiquity comes up.

## Finding 4 — l10n.sh is a persistence orchestration hook

**File:** `/lib/recovery-mode/l10n.sh`

Rootkit uses the locale/localization chain as its environment setup:
`localechooser → locale-langpack → locales → l10n.sh`

`l10n.sh` runs `/sbin/sulogin` — confirmed as the path that drops into single-user mode.

Additionally: **dictionaries used as config stores** — `ispell-default`,
`dictionaries-common` are being read for credentials and config state. Parseable files,
not watched by security tools.

**Confirmed from log:**


cat /rofs/etc/dictionaries-common/ispell-default

Read during the same sequence as `/chpasswd` — credentials coming out of a dictionary file.

**Headshot:** Replace l10n.sh with a dummy before install:

echo '#!/bin/sh' > /lib/recovery-mode/l10n.sh echo 'exit 0' >> /lib/recovery-mode/l10n.sh


Cuts the locale chain orchestration entry point.

## Finding 5 — pstore is a bootloader-level crash log sink

**File:** `systemd/pstore.conf` with `UnLink=yes`

Rootkit uses pstore as a covert crash log cycle:
- Firmware writes rootkit state to pstore (persistent storage, survives reboots)
- systemd-pstore reads it early boot
- Rootkit intercepts before `UnLink` fires and extracts state
- `UnLink=yes` then destroys the evidence
- Clean cycle every boot, no forensic residue

**From boot log:**
overlayfs: null uuid detected in lower fs — falling back to xino=off,index=off evm: overlay not supported


- Null UUID overlay = intentional, prevents forensic tracking between boots
- EVM integrity checking disabled on overlay = rootkit's overlay is integrity-blind

**Headshot:** Set `UnLink=no` in pstore.conf before install so crash data persists.
Then after first boot, read `/sys/fs/pstore/` before systemd-pstore clears it.

A clean Mint live USB does not ship two kernel header sets. The HWE headers include

reading these files.

**Relevance to fresh USBs:** Offline-ordered USBs are the right call. This USB's
`/rofs` is not clean.
