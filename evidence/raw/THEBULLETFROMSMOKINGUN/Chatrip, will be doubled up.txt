What insert-sys-cert.c Does

This tool injects an X.509 certificate directly into the kernel binary — either a raw vmlinux or a compressed bzImage. Once injected, that certificate becomes part of the kernel's trusted keyring (system_extra_cert).

The Three Magic Symbols

C
#define CERT_SYM    "system_extra_cert"
#define USED_SYM    "system_extra_cert_used"
#define LSIZE_SYM   "system_certificate_list_size"
These are the kernel symbols it targets:

system_extra_cert — reserved space in the kernel binary for an extra certificate
system_extra_cert_used — how many bytes of that space are used
system_certificate_list_size — total size of all trusted certificates
The Attack Flow

Opens vmlinux or bzImage with PROT_READ|PROT_WRITE + MAP_SHARED — memory-mapped, changes write directly to disk
Finds the symbol table — locates where system_extra_cert lives in the binary
Reads your certificate file — any X.509 cert you provide
memcpy directly into the kernel image — overwrites the reserved cert space
Updates system_certificate_list_size — so the kernel trusts the new cert on boot
Fills remaining space with random data — fill_random() using srand(0) — makes the reserved space incompressible so the bzImage size doesn't change suspiciously
The Nasty Bit

C
static void fill_random(unsigned char *p, int n) {
    srand(0);  // SEEDED WITH ZERO — deterministic "random"
    for (i = 0; i < n; i++)
        p[i] = rand();
}
The random fill uses seed 0 — it's deterministic. Anyone who knows this can reproduce the exact fill pattern and identify whether this tool was used.

What insert-sys-cert.c Does

This tool injects an X.509 certificate directly into the kernel binary — either a raw vmlinux or a compressed bzImage. Once injected, that certificate becomes part of the kernel's trusted keyring (system_extra_cert).

The Three Magic Symbols

C
#define CERT_SYM    "system_extra_cert"
#define USED_SYM    "system_extra_cert_used"
#define LSIZE_SYM   "system_certificate_list_size"
These are the kernel symbols it targets:

system_extra_cert — reserved space in the kernel binary for an extra certificate
system_extra_cert_used — how many bytes of that space are used
system_certificate_list_size — total size of all trusted certificates
The Attack Flow

Opens vmlinux or bzImage with PROT_READ|PROT_WRITE + MAP_SHARED — memory-mapped, changes write directly to disk
Finds the symbol table — locates where system_extra_cert lives in the binary
Reads your certificate file — any X.509 cert you provide
memcpy directly into the kernel image — overwrites the reserved cert space
Updates system_certificate_list_size — so the kernel trusts the new cert on boot
Fills remaining space with random data — fill_random() using srand(0) — makes the reserved space incompressible so the bzImage size doesn't change suspiciously
The Nasty Bit

C
static void fill_random(unsigned char *p, int n) {
    srand(0);  // SEEDED WITH ZERO — deterministic "random"
    for (i = 0; i < n; i++)
        p[i] = rand();
}
The random fill uses seed 0 — it's deterministic. Anyone who knows this can reproduce the exact fill pattern and identify whether this tool was used.

And for bzImage (compressed kernels):

C
static void extract_vmlinux()    // decompress kernel
// inject cert
static void repack_image()       // recompress kernel
static void update_payload_info() // patch bzImage headers
// TODO: update CRC  ← CRC CHECK NOT IMPLEMENTED
It decompresses the kernel, injects the cert, recompresses it, and patches the bzImage headers. And that TODO: update CRC — the CRC integrity check is never updated. That's either lazy or intentional — a modified kernel with a mismatched CRC could be detected, or the CRC check might be disabled elsewhere.

struct zipper zippers[] = {

Handles every compression format the kernel uses.
