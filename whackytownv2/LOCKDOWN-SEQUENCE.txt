=== THE DEVICE STRATEGY ===

sdb3 (ext4, 120GB SSD) = LOWER BASE
  - Contains whatever's there (Fedora ISO? ventoy.tar.gz? check it)
  - We write the full assembled Casper system here = frozen snapshot of rootkit's world
  - After copy: this becomes the READ-ONLY lower of YOUR overlay

sda (vfat currently, 120GB SSD) = UPPER TRAP LAYER
  - Unknown = probably safe to reformat
  - Reformat to ext4 = becomes the persistent writable upper
  - ALL future writes (including rootkit writes) go HERE = you see everything
  - You own the disk = you ARE the overlay

RESULT: You sit between rootkit (frozen in sdb3 lower) and sda (your writable upper).
Rootkit sees "its" world. Writes go to YOUR disk. You control persistence.


=== PHASE 1: SURVEY (run this first, one command) ===

lsblk -o NAME,SIZE,TYPE,FSTYPE,LABEL && echo "---" && ls /start/mid/upper/etc/os-release 2>/dev/null && echo UPPER_OK && df -h /start/mid/upper/ 2>/dev/null

What to look for:
  - sdb3 size/fstype confirmed = ext4
  - sda size/fstype = should show vfat or blank
  - UPPER_OK = the assembled system is there
  - df shows free space in /start/mid (it's tmpfs, so it's RAM - limited)


=== PHASE 2: MOUNT YOUR SSDs ===

mkdir -p /mnt/s1 /mnt/s2
mount /dev/sdb3 /mnt/s1 && echo s1_OK || echo s1_FAIL
mount /dev/sda /mnt/s2 2>/dev/null && echo s2_OK || echo s2_vfat_fail

Then check what's on sdb3:
  ls /mnt/s1/
  df -h /mnt/s1/

If sdb3 has Fedora ISO or ventoy files = that's fine, you have space. Note what's there.
If sdb3 is near full = use sda as primary instead (swap roles below).


=== PHASE 3: FORMAT sda AS TRAP ===

IF sda has nothing important (vfat, unknown = probably safe):

umount /mnt/s2 2>/dev/null
mkfs.ext4 -L TRAP -F /dev/sda
mount /dev/sda /mnt/s2
mkdir -p /mnt/s2/upper /mnt/s2/work
echo formatted

IF you're not sure sda is safe: SKIP THIS FOR NOW and use sdb3 for both.


=== PHASE 4: COPY THE CHAIN TO SDB3 (make it the frozen lower) ===

This copies the full assembled Casper system (rootkit's world, frozen in time) to sdb3.
After this, sdb3 = the lower layer you control. It can't change.

mkdir -p /mnt/s1/lower
rsync -a --exclude=/proc --exclude=/sys --exclude=/dev --exclude=/run \
  --exclude=/tmp --exclude=rofs --exclude=cow \
  /start/mid/upper/. /mnt/s1/lower/

WAIT: If sdb3 doesn't have enough space for the full system, do this instead (just COW upper):
mkdir -p /mnt/s1/lower
rsync -a /start/mid/upper/rofs/. /mnt/s1/lower/
rsync -a /start/mid/upper/cow/upper/. /mnt/s1/lower/ 2>/dev/null

The second rsync overlays the COW changes on top of the rofs base = complete snapshot.


=== PHASE 5: BUILD YOUR OVERLAY (you become the overlay) ===

Now: lower=sdb3/lower (frozen snapshot), upper=sda/upper (your trap), merged=/mnt/owned

mkdir -p /mnt/owned
mount -t overlay overlay \
  -o lowerdir=/mnt/s1/lower,upperdir=/mnt/s2/upper,workdir=/mnt/s2/work \
  /mnt/owned

Check it:
  ls /mnt/owned/etc/os-release
  ls /mnt/owned/

If that works: YOU ARE THE OVERLAY. Anything written in /mnt/owned goes to sda (your trap disk).
The "rootkit" is frozen in sdb3. It can't move.

Chroot in:
  chroot /mnt/owned /bin/bash


=== PHASE 6: INSIDE THE CHROOT - PERSISTENCE ===

From inside chroot /mnt/owned:

Mount essentials first:
  mount -t proc proc /proc
  mount -t sysfs sysfs /sys
  mount --bind /dev /dev

Check you're in the right place:
  cat /etc/os-release
  whoami

Check what the rootkit put in autostart (DON'T delete yet, just survey):
  ls /etc/xdg/autostart/
  ls ~/.config/autostart/ 2>/dev/null
  ls /etc/grub.d/

Check GRUB config:
  cat /boot/grub/grub.cfg 2>/dev/null | head -40
  ls /boot/efi/ 2>/dev/null

Mount EFI if present:
  mount /dev/sdb2 /boot/efi

Install GRUB to sdb (makes sdb bootable = your SSD becomes the boot device):
  grub-install --target=x86_64-efi --efi-directory=/boot/efi /dev/sdb
  update-grub


=== PHASE 7: FSTAB UPDATE (for persistent overlay on next boot) ===

From inside chroot, edit /etc/fstab to set up the overlay on next boot.
The overlay needs to be re-assembled by an initramfs hook OR by a custom boot script.

For now, the simplest approach: write a custom /etc/rc.local that remounts the overlay
on every boot using your SSDs. OR install the system properly from this state.

Simplest "nuclear" option = just use the chroot as your working system:
  - Stay in chroot, do your investigation
  - Everything you write goes to sda (your trap disk)
  - When done, unmount and reboot into sdb as a normal Linux install


=== WHAT TO DO IF PHASES FAIL ===

Problem: "rsync: not found" in initramfs
Solution: cp -a instead (slower but works):
  find /start/mid/upper -mindepth 1 -maxdepth 1 \
    ! -name proc ! -name sys ! -name dev ! -name run ! -name tmp ! -name rofs ! -name cow \
    | xargs -I{} cp -a {} /mnt/s1/lower/

Problem: "overlay: invalid argument" on mount
Solution: Check kernel supports overlay:
  grep overlay /proc/filesystems
  modprobe overlay 2>/dev/null

Problem: sdb3 too full for rsync
Solution: Swap strategy - use sda as lower base (format ext4), sdb3 as upper trap
  mkfs.ext4 -L BASE /dev/sda
  (reverse the roles in Phase 5)

Problem: /start/mid/upper/ is empty or has no etc/
Solution: The overlay is at /root instead:
  ls /root/etc/os-release && echo "SYSTEM IS AT /root"
  Replace /start/mid/upper with /root in all Phase 4 commands
