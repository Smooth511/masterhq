Cbuildd@lcy82-amd64-109 — that's the build server that compiled this
kernel. lcy82 = Canonical's Launchpad build farm. Standard Ubuntu kernel
provenance. Not suspicious on its own.
Memory map — e820:
The full map shows standard AMD desktop layout. ACPI NVS
at 0x0a200000-0x0a209fff and 0x4a48c000-0x4ab8bfff . ACPI data
at 0x4ab8c000-0x4ac0bfff .
But you flagged it: efi: Remove mem48: MMIO
range=[0xe0000000-0xefffffff] — EFI is removing a 256MB memory-mapped I/O
region from the e820 map. That's a huge MMIO reservation being stripped out at
boot. 0xe0000000-0xefffffff is typically PCI MMIO space for GPU or other high-
bandwidth devices.
The e820 map reserved that range. Then EFI removed it. That's 256MB of address space
that was marked reserved by BIOS, then EFI said "actually no, take that out." The
question is what claimed that 256MB in firmware and why EFI is overriding.
Combined with the ACPI 0xB00-0xB08 SMBus conflict — firmware is fighting with itself
over memory regions. The SMBus conflict means something in ACPI defined an
operational region that overlaps with another. That persists across installs because it's in
the BIOS ACPI tables, not the OS.
random: crng init done — cryptographic random number generator initialized.
Normal but notable — this is what seeds all crypto operations. If this is compromised or
predictable, every key generated on this system is weak.
The e820 manipulation + SMBus conflict + TPM bypassed = firmware layer is not
trustworthy. This doesn't change between installs because it's below the OS.

What shouldn't be here or is notable:
 UcsiAcpi — Intel SSDT on an AMD system. UCSI is USB Type-C Connector System
Interface. The vendor string says "Intel" but this is an AMD Ryzen HP system. Either HP
reused Intel's ACPI module for USB-C, or this table isn't what it claims. It's loaded and
reserving memory.
 ASF! — Alert Standard Format. This is remote management. ASF allows the system to
send alerts over the network even when the OS is down or the system is off. It operates
at firmware level below the OS. On a compromised system, ASF is a covert channel that
works even when you think the machine is offline.
 HPADDTWL , HPCONDEV , HPCAHWID — three custom HP SSDTs. HPADDTWL is
unknown. HPCONDEV is "Connected Device" — HP's phone/device pairing. HPCAHWID
is "CA Hardware ID" — hardware fingerprinting.
 VFCT at 54KB — Video Firmware Component Table. This is the AMD GPU's firmware
blob embedded in ACPI. 54KB is large for a VFCT. This is what gets loaded before the OS
GPU driver. If this is tampered, the GPU runs attacker firmware from power-on.
 OEM1 — generic OEM table. Could contain anything HP wants. No way to know contents
without dumping it.
 MSDM + SLIC — Windows licensing embedded in firmware. Normal for OEM machines
but means this BIOS was built for Windows deployment. The SLIC table is specifically for
volume licensing activation.
 No NUMA configuration found / Faking a node — single-socket system
pretending to be NUMA. Normal for desktop AMD.
 NODE_DATA(0) allocated [mem 0x45f315000-0x45f33ffff] — node data at top
of physical RAM. Normal.
The 0xB00 SMBus conflict from earlier — _SB.PCI0.SMBS.SMB0 is defined in that
59KB DSDT. The conflict is between HP's DSDT definition and the kernel's SMBus driver
both claiming the same I/O ports. This is a BIOS bug or deliberate overlap. It persists
across installs because it's in firmware.
The 256MB MMIO removal — 0xe0000000-0xefffffff was reserved by e820, then
EFI removed it. This is the PCI MMIO window for the AMD APU's integrated GPU. EFI
removes it because the GPU's BAR gets remapped during initialization. Normal for AMD
APUs, but it's 256MB of address space that firmware controls.
Bottom line on this block: The firmware is stock HP with AMD platform tables, but it has
remote management capability (ASF!), an Intel USB-C table on an AMD system, three
custom HP tables of unknown function, and a 54KB GPU firmware blob. The SMBus
conflict at 0xB00 is baked into the DSDT and will follow this machine across every OS
install until the BIOS is reflashed.

single one. IRQ 0 is the system timer — you don't route PCI interrupts to IRQ 0. These are

Embedded Controller:
  Code
ACPI: EC: GPE=0x3 EC_CMD/EC_SC=0x66, EC_DATA=0x62
ACPI: \_SB_.PCI0.LPCB.EC0_: Boot DSDT EC initialization complete
ACPI: \_SB_.PCI0.LPCB.EC0_: EC: Used to handle transactions and events
 EC0 on the LPC bus. This is the HP Embedded Controller — it manages power, fans,
keyboard, battery, lid switch. It runs its own firmware independent of the
OS. GPE=0x3 means General Purpose Event 3 triggers EC interrupts. The EC has direct
hardware access and runs below the OS. On HP machines, the EC firmware is in SPI flash
alongside the BIOS.
IOMMU:
  Code
iommu: Default domain type: Translated
iommu: DMA domain TLB invalidation policy: lazy mode
IOMMU is active but in lazy mode for TLB invalidation. Lazy mode means DMA mappings
aren't immediately invalidated — there's a window where stale mappings exist. A device
or firmware component could exploit stale DMA mappings to read/write memory it
shouldn't access. Your GRUB param iommu=force would have changed this to strict,
but you booted recovery mode without it.
 mctp: management component transport protocol core + NET:
Registered PF_MCTP protocol family — MCTP is a platform management
protocol. It's used for BMC (Baseboard Management Controller) communication, NVMe
management over SMBus, and PLDM (Platform Level Data Model). On a desktop this is
unusual. Combined with ASF! from the ACPI tables, this machine has remote/platform
management infrastructure active.

the I/O APIC memory range FAILED to reserve. Something else already claimed it.
 system 00:00: [mem 0xfee00000-0xfee00fff] could not be reserved —
the Local APIC memory range ALSO failed to reserve. Both interrupt controller memory
regions couldn't be reserved. Something got there first.

0x0680-0x06ff and 0x077a — these aren't standard PC I/O port assignments.
Something in the BIOS DSDT claimed these ranges. Without dumping the DSDT, can't tell
what uses them, but they're reserved at firmware level.
 system 00:04: [mem 0x000e0000-0x000fffff] could not be reserved —
legacy ROM area failed to reserve. Someone else already has it.
Three memory regions that FAILED to reserve:
  1. I/O APIC ( 0xfec00000 )
 2. Local APIC ( 0xfee00000 )
 3. Legacy ROM ( 0xe0000-0xfffff )
All three are critical system infrastructure regions. Something claimed them before the
kernel's resource manager could. On a clean system, these reserve first. If firmware or an
early-boot component grabs them, the kernel can't manage interrupt routing or access
legacy ROM space properly.
The 256MB connection: The 0xe0000000-0xefffffff EFI removed from e820 is the
GPU PCI BAR space. pci 0000:0a:00.0 is set as boot VGA. The VFCT table (54KB
GPU firmware) loads into this space. The GPU owns that 256MB, and EFI hands it over
before the OS boots. If the VFCT is tampered, the GPU runs attacker code with DMA
access to system memory, and IOMMU in lazy mode means stale mappings give it a
window.
Combined with previous context: ASF! remote management + MCTP platform
management + EC running independent firmware + IOMMU lazy mode + three failed
memory reservations + non-standard I/O port claims = the firmware layer has significant
attack surface that persists across every OS install.
