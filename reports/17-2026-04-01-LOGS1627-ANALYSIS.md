### THREE Different PCI Addresses (F1 — UPGRADED from "swap" to "rotation")

The NVMe controller for the Ubuntu install drive appears at **three different PCI addresses** across boots:

1. **`0000:02:00.0`** — Boots -6, -5, -4 (as nvme1 when paired with nvme0 at 05:00.0)
2. **`0000:04:00.0`** — Boots -3, -2, 0 (solo, nvme1 completely absent)
3. **`0000:05:00.0`** — Boots -6, -5, -4 (as nvme0 when paired with nvme1 at 02:00.0)

Real hardware has **fixed PCI addresses** determined by the physical PCIe slot. A drive cannot appear at 3 different addresses unless:
- A hypervisor is remapping PCI topology between boots
- The ACPI tables are being regenerated with different PCI assignments

### Second NVMe Disappears Entirely (F16)

On the 2097-RTC boots (dmesg.0, dmesg.1.gz) and the final stable boot (dmesg), **only one NVMe is visible**. The second drive (Samsung 970 EVO 256GB NVMe) vanishes completely. No PCI enumeration, no driver load, no partition detection. As detailed in §19.3, by the time of these logs the active Windows/yoink NTFS partition resides on the Intel SSD 665p 1TB device; the disappearing device here is the Samsung 256GB NVMe itself, not the current Windows/yoink volume.

This matches the hypervisor model: the attacker's hypervisor can present 1 or 2 virtual NVMe devices depending on the boot scenario. The "xen" directory on the NTFS partition (F4) and the EDK2 DMAR tables (F5) only appear when both drives are presented.

## 3. DMAR — VM FIRMWARE PROVIDER (F5)

```
dmesg.3.gz: ACPI: DMAR 0x000000006D304000 0000A8 (v01 INTEL  EDK2  00000002  01000013)
dmesg.3.gz: ACPI: DMAR 0x000000006D305000 0000A8 (v01 INTEL  EDK2  00000002  01000013)
dmesg.4.gz: ACPI: DMAR 0x000000006D304000 0000A8 (v01 INTEL  EDK2  00000002  01000013)
dmesg.4.gz: ACPI: DMAR 0x000000006D305000 0000A8 (v01 INTEL  EDK2  00000002  01000013)
```

**Two DMAR tables at adjacent memory addresses**, both with provider string `EDK2`. Real ASUS BIOS hardware would show a single DMAR with ASUS/AMI provider string. EDK2 is TianoCore/OVMF — the open-source UEFI firmware used in virtual machines.

The EDK2 DMAR tables **only appear on boots where both NVMe drives are visible** (dmesg.3.gz and dmesg.4.gz). On boots where only one NVMe is presented (dmesg, dmesg.0, dmesg.1.gz, dmesg.2.gz), no DMAR table appears at all. This confirms the hypervisor dynamically injects DMAR tables when presenting the two-drive configuration.

### ACPI Table Memory Relocation (F14, F15) — NEW

| Boot | XSDT Address | XSDT Size | Memory Range |
|------|-------------|-----------|--------------|
| dmesg (current) | `0x89246728` | `0xD4` (212 bytes = 20 entries) | `0x89xxxxxx` |
| dmesg.0 | `0x89246728` | `0xD4` (20 entries) | `0x89xxxxxx` |
| dmesg.1.gz | `0x89246728` | `0xD4` (20 entries) | `0x89xxxxxx` |
| dmesg.2.gz | `0x8935C728` | `0xD4` (20 entries) | `0x89xxxxxx` |
| **dmesg.3.gz** | `0x6D35C728` | **`0xCC`** (204 bytes = **19 entries**) | **`0x6Dxxxxxx`** |
| **dmesg.4.gz** | `0x6D35C728` | **`0xCC`** (19 entries) | **`0x6Dxxxxxx`** |

The ACPI tables **move ~0x1C000000 (448MB) lower in memory** on the boots where EDK2 DMAR tables are present. Simultaneously, the XSDT table shrinks from 20 entries to 19, while **adding** the 2 DMAR entries. This means another table was removed to make room for the injected DMAR tables — the hypervisor is editing the ACPI table set dynamically.

On real hardware, ACPI tables live at fixed firmware-assigned addresses that don't change between reboots.
