### FINDING 1: Self-Signed CN=grub MOK Certificate in UEFI NVRAM

**Status: ✅ CONFIRMED — EVIDENCE IS REAL**  
**Assessment: 🔴 CRITICAL — with nuance**

The certificate exists. It was captured directly from `mokutil --db` output during a live forensic session. The certificate details are:

| Field | Value | Assessment |
|-------|-------|------------|
| Subject/Issuer | CN=grub (self-signed) | ⚠️ Ubuntu CAN auto-generate CN=grub certs — but see below |
| Not Before | Feb 24 22:38:00 2019 GMT | 🔴 **7 years before March 2026 install** — NOT generated by current install |
| Not After | Feb 21 22:38:00 2029 GMT | ⚠️ 10-year validity is common for self-signed MOK certs |
| CA:TRUE | Yes | 🔴 Standard Ubuntu MOK certs DO have CA:TRUE |
| Netscape Cert Type | ALL types enabled (SSL Client, SSL Server, S/MIME, Object Signing, SSL CA, S/MIME CA, Object Signing CA) | 🔴 **NOT STANDARD** — Ubuntu auto-generated certs use modern X.509 extensions, not legacy Netscape Cert Type with ALL flags |
| Extended Key Usage | Code Signing | ✅ Standard for MOK |
| SKI public footprint | Zero results worldwide | 🔴 A legitimate cert from a standard Ubuntu install WOULD appear in package contents / Launchpad build artifacts |

**Critical nuance the UEFI report should acknowledge:**

Ubuntu's `shim-signed` package **does** auto-enroll CN=grub MOK certificates. This is standard behavior. However:

1. **The creation date disproves auto-generation by the current install.** Auto-generated MOK certs have their "Not Before" date set to generation time. Feb 2019 ≠ March 22, 2026.

2. **The excessive Netscape Cert Type flags are non-standard.** Ubuntu's tools generate certs with modern X.509 extensions (Key Usage, Extended Key Usage, Basic Constraints). The full Netscape Cert Type field with ALL types enabled is characteristic of a **manually created cert using OpenSSL with a broad configuration**, not an auto-generated shim cert.

3. **The zero public footprint of the SKI hash is expected for ANY per-machine MOK cert** (auto-generated or attacker-created). This alone does not prove malice — it confirms uniqueness.

**Possible legitimate explanation:** If this machine had a previous Linux install in Feb 2019 (HP EliteDesk 705 G4 DM was released 2018-2019), that install could have enrolled this cert. It would survive in NVRAM across every subsequent OS install.

**Why this explanation is unlikely given the full evidence chain:** The DISM+Synergy evidence (Finding 8) proves human-in-the-loop control during OS deployment. That same actor had the access and opportunity to enroll a MOK certificate. The cert's excessive capabilities, combined with all other persistence indicators, strongly suggest deliberate attacker enrollment rather than a forgotten legitimate install.

**VERDICT: CONFIRMED REAL. Probably malicious given full evidence chain. Definitive proof requires determining if a legitimate Linux install existed on this machine in Feb 2019.**

### FINDING 4: mokutil --list-enrolled Selectively Blocked

**Status: ✅ CONFIRMED — multiple attempts documented**  
**Assessment: 🔴 CRITICAL**

The user (running as root) executed `mokutil --list-enrolled` multiple times across the session. Every time, it printed the full help text instead of listing enrolled keys. Meanwhile, `mokutil --db` worked correctly and returned certificate data.

These commands access different EFI variable stores:
- `--db` reads the UEFI Secure Boot db variable
- `--list-enrolled` reads the MokListRT variable

**Possible explanations:**
1. **Modified mokutil binary** — the binary was patched to suppress MOK enumeration
2. **EFI variable permissions** — MokListRT has been locked at the firmware level
3. **mokutil version/syntax issue** — unlikely given repeated attempts with correct syntax as root
4. **Shim configuration** — some shim versions restrict MOK variable access

The selective nature (db works, MOK doesn't) is the strongest indicator. If this were a general EFI variable access problem, both would fail.

**VERDICT: CONFIRMED. Active interference with forensic investigation is the most likely explanation, but other agents should attempt to reproduce this on a clean machine with the same Ubuntu/mokutil version to rule out a known bug.**

### FINDING 5: EFI Memory Map Changes Between Cold Boots

**Status: ✅ CONFIRMED — from journal evidence**  
**Assessment: 🟡 MEDIUM-HIGH**

Boot 2 showed 10 additional MMIO entries, including the SPI flash/BIOS ROM range (`0xff000000–0xffffffff`). Kernel setup_data shifted by ~132KB.

**Counter-argument:** On a machine with no CMOS battery, cold boot behavior can be less deterministic. UEFI firmware re-enumerates devices from scratch, and minor differences in USB/device initialization timing can produce slightly different memory maps.

**Supporting argument:** The SPI flash range appearing in Boot 2 is significant — this is the BIOS ROM address space. Its removal from the e820 map means firmware is actively managing (or hiding) this region. On identical hardware with identical boot conditions, this should not change.

**VERDICT: CONFIRMED as real observation. Significance depends on whether this machine has consistent boot conditions between the two captures. The SPI flash range appearance is the most concerning element.**
