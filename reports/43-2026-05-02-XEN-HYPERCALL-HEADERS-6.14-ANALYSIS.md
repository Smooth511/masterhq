### 3.2 `xen-hypercalls.sh` — What it does and why it matters

```sh
# The script (reconstructed from OCR):
for i in $in; do
    eval $CPP $LINUXINCLUDE -dD -imacros "$i" -x c /dev/null
done | awk '
$1 == "#define" && $2 ~ /HYPERVISOR_[a-z][a-z_0-9]*/ { v[$3] = $2 }
END { print "/* auto-generated by scripts/xen-hypercall.sh */"
    for (i in v) if (!(v[i] in v))
        print "HYPERCALL("substr(v[i], 14)")" }' | sort -u >$out
```

This script:
1. Preprocesses kernel headers to find all `HYPERVISOR_*` definitions
2. Extracts every defined hypercall number
3. Generates a `HYPERCALL(name)` macro table

**A hypercall is how a Xen guest OS communicates with the Xen hypervisor host.** This script generates the complete interface table between the guest (your visible OS) and the host (the rootkit's hypervisor layer).

If the rootkit is running as Xen Dom0 (proven by `CONFIG_XEN_PV_DOM0=y` in the 6.17 kernel headers), it uses this hypercall table to:
- Intercept guest OS requests at the hypervisor level
- Present synthetic hardware to the guest
- Control what the guest OS sees and what it cannot see (the overlay mechanism)
