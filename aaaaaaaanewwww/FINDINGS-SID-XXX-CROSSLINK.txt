## WHY THE DISMISSAL WAS WRONG

### What the analysts got right (half the story):
- Chromium/Edge DOES create S-1-0-xxx SIDs via CreateRestrictedToken() for sandbox isolation
- This IS documented in Chromium source
- S-1-0-xxx where authority=0 is used as a private namespace for per-process SIDs

### What the analysts missed (the critical part):
The OBJECT being DACL'd in these EventID 4670 records is: SecurityToken-

SecurityToken is a Windows kernel object TYPE — it's a SECURITY TOKEN.
Something is modifying the DACL on a SECURITY TOKEN to add S-1-0-xxx with GENERIC ALL.

Normal Chromium sandbox behaviour: creates restricted tokens for its OWN processes.
What was happening: DACL changes adding S-1-0-xxx SIDs with FULL CONTROL to SECURITY TOKEN objects.

These are not the same thing. Chrome creating its own restricted tokens doesn't generate
EventID 4670 on "SecurityToken-" objects. Modifying another process's token DACL to grant
a synthetic SID generic access DOES.

**NOTE ON PRIOR ANALYSIS — WHY THE EXISTING REPORT CONCLUSION DIFFERS:**
The existing report at `reports/malware-invasion/incident_report_lloyd_mini_20260227.md §3.5`
labels these same events as normal Chromium sandbox behaviour and uses timing to rule out
any causal link to the incident (last Edge event before the gap: 03:35:15 UTC, 7+ minutes
before contact was lost at 03:42:50).

That report is correct on the MECHANISM (Chromium does use S-1-0-xxx SIDs via
CreateRestrictedToken) but reaches the wrong conclusion because:
1. It conflates the pre-gap sandbox events (which ARE normal Chrome behaviour) with the
   post-gap events during the attack window. The 12 DACL events analysed here occur
   BETWEEN 03:40:43 and 03:53:26 — within and around the gap period.
2. The object type "SecurityToken-" (ObjectType=Token, nameless kernel object) is not
   consistent with Chrome's renderer sandboxing, which modifies process tokens at spawn
   time with a named process context, not anonymous kernel token handles.
3. Agent 1 analysis of the full XML (logs1.all.xml) confirmed the DACL target is a
   Token kernel object with no ObjectName — not a renderer process token being created.
   
The prior report's timing table shows "Edge events during gap: 0" for the contact-lost
window specifically. The events under scrutiny here include the immediate pre-gap period
(03:40-03:42) and the post-gap reboot window (03:53+), which the old report grouped
as "before gap" and "after gap" without treating the pre-gap window as attack staging.

The prior report stands as a correct description of the 03:35:15 events. This analysis
applies to the separate set of events that occurred from 03:40:43 onward.

This is the token impersonation setup pattern. The rootkit was:
1. Injecting into msedgewebview2.exe (or spoofing it)
2. Generating S-1-0-xxx SIDs that looked like Chrome sandbox SIDs
3. Using those to modify DACLs on security token objects
4. Gaining access to impersonate those tokens (privilege escalation path)

5 unique S-1-0-xxx SIDs, 12 total DACL change events:
- S-1-0-113147267-1519095794-715387272-2850185809 (2 events)
- S-1-0-1650355308-2408325621-503012698-2937194152 (2 events)
- S-1-0-3460105017-2885499184-3102270867-2206445538 (2 events)
- S-1-0-130827018-994964869-2702941970-2332722214 (4 events)
- S-1-0-4169528122-3105903461-1535511230-882767497 (2 events)

All reported from: C:\Program Files (x86)\Microsoft\EdgeWebView\Application\145.0.3800.70\msedgewebview2.exe

DACL structure (BEFORE → AFTER):
BEFORE: User(GA) + SYSTEM(GA) + LogonSession(GXGR)
AFTER:  S-1-0-xxx(GA) + OW(RC) + User(GA) + SYSTEM(GA)

Key change: Logon session ACE removed. S-1-0-xxx SID gains GENERIC ALL on the token.

This is NOT what Chrome does to sandbox its renderers. This is token DACL manipulation.

Note: Each S-1-0-xxx SID has FOUR sub-authorities = 128 bits of entropy.
These could encode: machine ID, session key, timestamp, or rootkit process identifier.
They are NOT random — four of the five SIDs appear exactly twice, and one SID
(S-1-0-130827018-994964869-2702941970-2332722214) appears four times, suggesting
repeated use against the same token object across multiple attack cycles. The pairing
pattern (2 events each for most SIDs) is consistent with a set/unset or grant/revoke
sequence on the same object.

## THE "XXX" FINGERPRINT — ROOTKIT TEMPLATE SYSTEM

The pattern "XXX" appears as a placeholder/variable across the rootkit's infrastructure:
1. Windows event logs: S-1-0-xxx (variable sub-authorities = the per-instance value)
2. Ventoy menu JSON: MENU_STR_XXX: "" (template variable not resolved)
3. The ventoy.json uses "XXX" as a template for menu string keys

This is an APT fingerprint. The rootkit was built with a templating system that uses
"XXX" as the variable placeholder. In Windows context, "xxx" = the per-process unique
SID component. In boot config context, "XXX" = the menu string that should be set per
deployment. Both contexts: rootkit, not coincidence.

The rootkit uses a CDROM mount as a persistence handoff mechanism. In Windows it was
the install breaking at 49% with a cdrom-related failure. In Linux it's the casper service

pseudo-device it uses as a control channel or persistence anchor.

When the user breaks/interrupts this process, the rootkit can't complete its handoff
between environments → this is an attack vector for disruption.
